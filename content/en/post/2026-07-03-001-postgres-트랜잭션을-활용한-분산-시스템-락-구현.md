# Implementing Distributed System Locks Using Postgres Transactions

I recently read an interesting post on [Hacker News](https://news.ycombinator.com/) making the claim: "Postgres transactions are a distributed systems superpower." In distributed systems, we often introduce dedicated systems like ZooKeeper or etcd for concurrency control. However, we might already be using a powerful ACID-compliant database. In this post, we will explore how to implement a stable distributed lock using advanced PostgreSQL features, without the complexity of external systems.

## Why Use a DB as a Lock Mechanism?

While working on the [ZeroClaw](/tags/zeroclaw) project, I needed to prevent resource conflicts between multiple agents. A simple `Mutex` in 'application memory' is useless in an environment with multiple server instances. I could have used Redis, but that would increase infrastructure complexity and make data persistence guarantees difficult.

On the other hand, using PostgreSQL offers the following advantages:
1.  **Data Integrity Guarantee**: Transactions can be rolled back, and locks are automatically released upon failure (connection loss detection).
2.  **Infrastructure Simplification**: We can leverage existing DB resources without maintaining a separate coordinator (Cluster).
3.  **Robust Isolation Levels**: We can safely prevent race conditions through mechanisms like Serializable Read Isolation.

## Core Design: Advisory Lock vs. Table Lock

Postgres offers two approaches:

1.  **Advisory Locks (`pg_advisory_lock`)**: These are locks used consensually between sessions. They are very fast, but if an application bug fails to release a lock, it's difficult for DB administrators to check and release it externally.
2.  **Table-Based Lock (Pessimistic Locking)**: This uses the `SELECT ... FOR UPDATE SKIP LOCKED` syntax. It's intuitive because it uses standard SQL, and it's easy to monitor the state of DB tables.

We will adopt the table-based lock approach for **transparency and ease of management**. In particular, the `SKIP LOCKED` syntax is optimized for processing tasks in a queue-like manner because it skips rows that are locked and immediately finds the next row.

## Implementation: Rust Meets SQL

We will implement this in a [Rust](/tags/rust) environment. The hypothetical scenario is: "When an agent is assigned a task, it uniquely claims one task to avoid duplication with other agents."

### 1. Defining the Table Schema

First, let's create a simple table to store tasks.

```sql
CREATE TABLE agent_tasks (
    id SERIAL PRIMARY KEY,
    payload JSONB NOT NULL,
    status VARCHAR(20) DEFAULT 'pending', -- pending, processing, done
    locked_at TIMESTAMP,
    locked_by VARCHAR(100)
);

-- Index creation is mandatory (for performance improvement)
CREATE INDEX idx_tasks_status ON agent_tasks(status);
```

### 2. Lock Acquisition Logic (Rust & SQLx)

Now, let's write the logic to acquire the lock in Rust code. The key here is to **update the status and return the row within a single query**.

```rust
use sqlx::PgPool;
use serde_json::Value;

// Struct definition
#[derive(Debug)]
struct Task {
    id: i32,
    payload: Value,
}

/// Fetches a task and acquires a lock.
/// Returns None if a lock cannot be acquired.
async fn claim_task(pool: &PgPool, worker_id: &str) -> Result<Option<Task>, sqlx::Error> {
    let mut tx = pool.begin().await?;

    let result = sqlx::query_as::<_, Task>(
        r#"
        UPDATE agent_tasks 
        SET 
            status = 'processing', 
            locked_at = NOW(), 
            locked_by = $1
        WHERE id = (
            SELECT id 
            FROM agent_tasks 
            WHERE status = 'pending' 
            FOR UPDATE SKIP LOCKED 
            LIMIT 1
            FOR UPDATE
        )
        RETURNING id, payload
        "#
    )
    .bind(worker_id)
    .fetch_optional(&mut *tx)
    .await?;

    tx.commit().await?;
    Ok(result)
}
```

#### Code Analysis
*   **`FOR UPDATE SKIP LOCKED`**: This is the magic spell. It finds a row where `status = 'pending'`, but if another transaction has already locked it (Locked), it simply skips it (Skip). Therefore, the agent doesn't wait for the lock; it immediately finds the next available task or receives an empty result.
*   **Transaction (`tx`)**: The update and return must be processed atomically.

### 3. Lock Release and Timeout Handling

When the task is completed, its status must be changed. What if the application crashes and fails to change the status?

```rust
async fn complete_task(pool: &PgPool, task_id: i32, result: Value) -> Result<(), sqlx::Error> {
    sqlx::query(
        r#"UPDATE agent_tasks SET status = 'done', payload = $2 WHERE id = $1"#
    )
    .bind(task_id)
    .bind(result)
    .execute(pool)
    .await?;
    Ok(())
}

// For fault tolerance: A periodic job to reset tasks that have been in 'processing' status for too long
async fn reclaim_stale_tasks(pool: &PgPool) -> Result<(), sqlx::Error> {
    sqlx::query(
        r#"UPDATE agent_tasks 
           SET status = 'pending', locked_by = NULL, locked_at = NULL 
           WHERE status = 'processing' 
           AND locked_at < NOW() - INTERVAL '5 minutes'"#
    )
    .execute(pool)
    .await?;
    Ok(())
}
```

## Points to Note and Limitations

This method is not a silver bullet. You need to consider the following:

1.  **DB Load**: If lock acquisition attempts are very frequent (e.g., thousands per second), it can put a load on the DB. In this case, Redis might be a better choice.
2.  **Network Latency**: If the network communication with the DB is remote, lock acquisition will inevitably be slower than local memory locks.

## Conclusion

When building systems like [Discord MCP](/tags/discord-mcp) or [Cloud Monitor](/tags/cloud-monitor), it's wise to minimize technical debt unless building at a massive scale. PostgreSQL's `FOR UPDATE SKIP LOCKED` is one of the most elegant and practical ways to implement distributed locks. Apply it to your code today without the need for separate infrastructure setup.
