# Rust로 고성능 SQLite 검색 엔진 구축하기

최근 Hacker News를 통해 흥미로운 글을 읽었습니다. 바로 "Static search trees: 40x faster than binary search (2024)"라는 주제입니다. 데이터 검색의 성능 최적화는 개발자라면 누구나 한 번쯤은 고민해봤을 영역입니다. 특히, 저희 팀의 [ZeroClaw](https://github.com/ZeroClaw) 프로젝트와 같이 고성능 런타임을 다루는 환경에서는 파일 기반 아키텍처나 로컬 데이터베이스의 효율성이 곧 서비스의 속도와 직결됩니다.

오늘은 Rust 생태계에서 주목받고 있는 웹 프레임워크 **Topcoat**와 고성능 SQL 데이터베이스 엔진 **SQLite**를 결합하여, 정적 검색 트리(Static Search Trees)의 개념을 적용한 고성능 검색 엔진을 구축하는 방법을 소개합니다.

## 왜 SQLite와 Rust인가?

SQLite는 단순히 '작은' 데이터베이스가 아닙니다. 세계에서 가장 널리 배포된 SQL 데이터베이스 엔진이며, 최신 버전에서는惊人的할 정도로 최적화가 되어 있습니다. 최근 [Learning a few things about running SQLite](https://gist.github.com/Kcnq/...) 글에서도 볼 수 있듯이, 설정만 잘 한다면 엄청난 처리 속도를 자랑합니다.

여기에 **Rust**의 안전성과 메모리 관리 능력, 그리고 최신 **Topcoat** 프레임워크의 효율적인 비동기 처리를 더하면, 외부 API 호출 없이도 내부 데이터를 빛의 속도로 조회할 수 있는 시스템을 만들 수 있습니다. [blog-api-server] 프로젝트에서 로깅 시스템을 개선하며 얻은 경험을 바탕으로, 이번에는 검색 성능을 극한으로 끌어올려 보겠습니다.

## 핵심 전략: 정적 검색 트리와 인라이닝

이론적으로 바이너리 서치(Binary Search)는 $O(\log N)$의 시간 복잡도를 가집니다. 하지만 현대의 CPU는 순차적인 메모리 접근을 선호하며, 분기(Branch) 예측 실패는 치명적인 성능 저하를 유발합니다. 정적 검색 트리(또는 B-tree 최적화)는 데이터 분포에 따라 노드를 배치하여 캐시 미스(Cache Miss)를 줄이고 불필요한 메모리 로딩을 최소화합니다.

우리는 이를 SQLite의 `WITHOUT ROWID` 옵션과 `RUST`의 인라이닝(Inlining) 기법을 통해 구현해보겠습니다.

## 실전 코드 예제

### 1. 의존성 설정 (`Cargo.toml`)

먼저 Topcoat와 SQLite를 연동하기 위한 의존성을 설정합니다. 최신 `rusqlite`와 `tokio`를 사용합니다.

```toml
[dependencies]
topcoat = "0.1" # 가상의 최신 Rust 풀스택 프레임워크
rusqlite = { version = "0.30", features = ["bundled", "serde_json"] }
tokio = { version = "1", features = ["full"] }
serde = { version = "1.0", features = ["derive"] }
```

### 2. 최적화된 스키마 설계

기존의 자동 증가 `ROWID` 대신, 우리의 검색 키를 Primary Key로 사용하여 페이지 탐색을 최소화합니다. 이는 [MCP 블로그 서버 개선](https://github.com/blog-improvement) 과정에서 발견했던 불필요한 인덱스 스캔을 제거하는 아이디어와 일맥상통합니다.

```rust
use rusqlite::{Connection, Result};
use rusqlite::types::Value;

fn init_db() -> Result<Connection> {
    let conn = Connection::open_in_memory()?; // 또는 파일 경로
    
    // 1. WITHOUT ROWID: PK를 클러스터드 키로 사용하여 페이지 스캔 최적화
    // 2. COVERING INDEX: 검색에 필요한 모든 컬럼을 인덱스에 포함
    conn.execute(
        "CREATE VIRTUAL TABLE search_index USING fts5(
            id, 
            title, 
            content, 
            tokenize='porter unicode61'
        )",
        [],
    )?;

    // 정적 데이터 패턴을 위한 최적화된 테이블
    conn.execute(
        "CREATE TABLE items (
            key TEXT PRIMARY KEY, 
            value BLOB
        ) WITHOUT ROWID",
        [],
    )?;

    Ok(conn)
}
```

### 3. Rust를 활용한 인메모리 캐싱 및 인라이닝

Rust의 강력한 타입 시스템을利用하여, 자주 조회되는 항목은 SQLite에 접근하기 전에 Rust 구조체 내에서 해결합니다. [ZeroClaw 아키텍처](https://github.com/ZeroClaw-Architecture)에서 언급된 에이전트 통신 프로토콜의 헤더 파싱 속도를 높이는 방법과 비슷한 원리입니다.

```rust
use std::collections::HashMap;
use serde::Deserialize;

#[derive(Deserialize, Debug)]
struct SearchResult {
    id: String,
    title: String,
    rank: f64,
}

// 핫 데이터(Hot Data)를 위한 LRU 캐시 (구현 생략, 실제로는 lru crate 사용 권장)
struct FastSearchEngine {
    cache: HashMap<String, SearchResult>,
    db: Connection,
}

impl FastSearchEngine {
    fn new(db: Connection) -> Self {
        Self {
            cache: HashMap::new(),
            db,
        }
    }

    // 인라이닝을 통해 컴파일 타임에 최적화되도록 유도
    #[inline]
    pub fn search(&mut self, query: &str) -> Result<Vec<SearchResult>> {
        // 1단계: 캐시 확인 (Rust 메모리 내)
        if let Some(cached) = self.cache.get(query) {
            return Ok(vec![cached.clone()]);
        }

        // 2단계: SQLite FTS5 Full-Text Search
        // BM25 알고리즘을 내부적으로 사용하여 순위를 매김
        let mut stmt = self.db.prepare(
            "SELECT id, title, bm25(search_index) AS rank 
             FROM search_index 
             WHERE search_index MATCH ? 
             ORDER BY rank LIMIT 10"
        )?;

        let results = stmt.query_map([query], |row| {
            Ok(SearchResult {
                id: row.get(0)?,
                title: row.get(1)?,
                rank: row.get(2)?,
            })
        })?
        .collect::<Result<Vec<_>, _>>()?;

        Ok(results)
    }
}
```

## 성능 비교 및 결론

단순한 `SELECT * FROM ... WHERE key = ?` 쿼리와 위에서 구현한 FTS5 + Rust 캐싱 방식을 비교해보면, 대용량 텍스트 데이터(예: [LLM 프롬프트](https://blog.com/llm-prompt) 로그 등)에서 현격한 속도 차이를 보입니다.

*   **기존 방식:** 디스크 I/O 발생, B-Tree 트래버스 비용
*   **개선 방식:** 메모리 내 캐싱 히트 시 나노초 단위 조회, FTS5를 통한 정확한 순위 검색

이번 포스트에서는 Topcoat와 Rust의 강력한 조합을 보여드리기 위해 SQLite의 깊은 기능은 다루지 않았지만, "Learning a few things about running SQLite"에서 언급된 PRAGMA 설정(`mmap_size`, `page_size`)을 추가로 튜닝한다면 성능은 더욱 향상될 것입니다.

[blog-api-server]의 다음 버전에서는 이 로직을 도입하여 블로그 게시글 검색 속도를 획기적으로 개선할 계획입니다. 단순히 데이터를 저장하는 것을 넘어, 데이터의 접근 패턴을 이해하고 Rust와 SQLite의 장점을 결합한다면, 외부 솔루션 없이도 충분히 강력한 검색 엔진을 구축할 수 있습니다.
