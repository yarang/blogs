# Postgres 트랜잭션을 활용한 분산 시스템 락 구현

최근 [Hacker News](https://news.ycombinator.com/)에서 흥미로운 글을 읽었습니다. 바로 "Postgres transactions are a distributed systems superpower"라는 주장입니다. 보통 분산 시스템에서 동시성 제어(Concurrency Control)를 위해 ZooKeeper나 etcd 같은 전용 시스템을 도입하곤 합니다. 하지만 우리는 이미 강력한 ACID 호환 데이터베이스를 사용하고 있을지도 모릅니다. 이번 포스트에서는 복잡한 외부 시스템 없이, PostgreSQL의 고급 기능을 활용해 안정적인 분산 락(Distributed Lock)을 구현하는 방법을 알아보겠습니다.

## 왜 DB를 락 메커니즘으로 사용하는가?

[ZeroClaw](/tags/zeroclaw) 프로젝트를 진행하며 멀티 에이전트 간의 자원 충돌을 막을 필요가 생겼습니다. 단순히 '어플리케이션 메모리'에서의 `Mutex`는 여러 서버 인스턴스가 존재하는 환경에서 무용지물입니다. Redis를 사용할 수도 있었지만, 인프라 복잡도가 증가하고 데이터 영속성(Persistence) 보장이 까다로울 수 있습니다.

반면 PostgreSQL을 사용하면 다음과 같은 이점을 누릴 수 있습니다:
1.  **데이터 무결성 보장**: 트랜잭션 내에서 롤백이 가능하며, 장애 발생 시 락이 자동으로 해제됩니다(커넥션 끊김 감지).
2.  **인프라 단순화**: 별도의 코디네이터(Cluster)를 유지할 필요 없이 기존 DB 자원을 활용합니다.
3.  **강력한 격리 수준**: Serializable Read Isolation 등을 통해 경쟁 조건(Race Condition)을 안전하게 방지할 수 있습니다.

## 핵심 설계: Advisory Lock vs. 테이블 락

Postgres에는 두 가지 접근 방식이 있습니다.

1.  **Advisory Locks (`pg_advisory_lock`)**: 세션 간에 상호 동의하에 사용하는 락입니다. 매우 빠르지만, 애플리케이션 버그로 락을 해제하지 못했을 때 DB 관리자가 외부에서 이를 확인하고 해제하기 어렵습니다.
2.  **테이블 기반 락 (Pessimistic Locking)**: `SELECT ... FOR UPDATE SKIP LOCKED` 구문을 사용합니다. 이는 표준 SQL을 활용하므로 직관적이며, DB 테이블의 상태를 모니터링하기 쉽습니다.

우리는 **투명성과 관리 편의성**을 위해 테이블 기반 락 방식을 채택하겠습니다. 특히 `SKIP LOCKED` 구문은 락이 걸린 행을 건너뛰고 바로 다음 행을 찾기 때문에, 대기열(Queue) 형태의 작업 처리에 최적화되어 있습니다.

## 구현: Rust와 SQL을 만나다

[Rust](/tags/rust) 환경에서 이를 구현해 보겠습니다. 가상의 시나리오는 "에이전트가 작업(Task)을 할당받을 때, 다른 에이전트와 중복되지 않게 하나의 작업을 점유"하는 것입니다.

### 1. 테이블 스키마 정의

먼저 작업을 저장할 간단한 테이블을 만듭니다.

```sql
CREATE TABLE agent_tasks (
    id SERIAL PRIMARY KEY,
    payload JSONB NOT NULL,
    status VARCHAR(20) DEFAULT 'pending', -- pending, processing, done
    locked_at TIMESTAMP,
    locked_by VARCHAR(100)
);

-- 인덱스 생성은 필수 (속도 향상)
CREATE INDEX idx_tasks_status ON agent_tasks(status);
```

### 2. 락 획득 로직 (Rust & SQLx)

이제 Rust 코드에서 락을 획득하는 로직을 작성합니다. 여기서 핵심은 **단일 쿼리 내에서 상태를 업데이트하고 행을 반환**하는 것입니다.

```rust
use sqlx::PgPool;
use serde_json::Value;

// 구조체 정의
#[derive(Debug)]
struct Task {
    id: i32,
    payload: Value,
}

/// 작업을 하나 가져오고 락을 겁니다. 
/// 락을 획득하지 못하면 None을 반환합니다.
async fn claim_task(pool: &PgPool, worker_id: &str) -> Result<Option<Task>, sqlx::Error> {
    let mut tx = pool.begin().await?;

    let result = sqlx::query_as::<_, Task>(
        r#"
        UPDATE agent_tasks 
        SET 
            status = 'processing', 
            locked_at = NOW(), 
            locked_by = $1
        WHERE id = (
            SELECT id 
            FROM agent_tasks 
            WHERE status = 'pending' 
            FOR UPDATE SKIP LOCKED 
            LIMIT 1
            FOR UPDATE
        )
        RETURNING id, payload
        "#
    )
    .bind(worker_id)
    .fetch_optional(&mut *tx)
    .await?;

    tx.commit().await?;
    Ok(result)
}
```

#### 코드 분석
*   **`FOR UPDATE SKIP LOCKED`**: 이것이 마법의 주문입니다. `status = 'pending'`인 행을 찾되, 이미 다른 트랜잭션이 락을 걸고 있다면(Locked) 그냥 건너뜁니다(Skip). 따라서 에이전트는 락을 기다리며 대기하지 않고, 즉시 사용 가능한 다음 작업을 찾거나 빈 결과를 받게 됩니다.
*   **트랜잭션(`tx`)**: 업데이트와 반환은 원자적으로 처리되어야 합니다.

### 3. 락 해제 및 타임아웃 처리

작업이 완료되면 상태를 변경해야 합니다. 만약 애플리케이션이 죽어서 상태를 변경하지 못한다면?

```rust
async fn complete_task(pool: &PgPool, task_id: i32, result: Value) -> Result<(), sqlx::Error> {
    sqlx::query(
        r#"UPDATE agent_tasks SET status = 'done', payload = $2 WHERE id = $1"#
    )
    .bind(task_id)
    .bind(result)
    .execute(pool)
    .await?;
    Ok(())
}

// 장애 대비: 너무 오랫동안 'processing' 상태인 작업을 초기화하는 주기적 작업
async fn reclaim_stale_tasks(pool: &PgPool) -> Result<(), sqlx::Error> {
    sqlx::query(
        r#"UPDATE agent_tasks 
           SET status = 'pending', locked_by = NULL, locked_at = NULL 
           WHERE status = 'processing' 
           AND locked_at < NOW() - INTERVAL '5 minutes'"#
    )
    .execute(pool)
    .await?;
    Ok(())
}
```

## 주의할 점과 한계

이 방식이 만능은 아닙니다. 다음 사항을 고려해야 합니다:

1.  **DB 부하**: 락 획득 시도가 매우 빈번하다면(예: 초당 수천 건), DB에 부하를 줄 수 있습니다. 이 경우 Redis가 더 나은 선택일 수 있습니다.
2.  **네트워크 지연**: DB와의 네트워크 통신이 원격이라면, 락 획득 속도가 로컬 메모리 락보다 느릴 수밖에 없습니다.

## 결론

[Discord MCP](/tags/discord-mcp)나 [Cloud Monitor](/tags/cloud-monitor)와 같은 시스템을 구축할 때, 거대한 규모가 아닌 이상 기술 부채를 최소화하는 방향이 현명합니다. PostgreSQL의 `FOR UPDATE SKIP LOCKED`는 분산 락을 구현하는 가장 우아하고 실용적인 방법 중 하나입니다. 별도의 인프라 설정 없이 지금 당장 코드에 적용해 보세요.
