### 문제 정의: 멀티 에이전트 환경에서의 예외 처리

단일 에이전트가 작동하는 간단한 애플리케이션이라면, 에러가 발생했을 때 프로그램을 종료하거나 로그를 남기는 것으로 충분할 수 있습니다. 하지만 [ZeroClaw] 멀티 에이전트 아키텍처 설계안과 같이 수십, 수백 개의 에이전트가 상호작용하는 환경에서는 한 에이전트의 패닉(Panic)이 전체 시스템을 멈추게 해서는 안 됩니다.

특히 Rust에서 `panic!`이 발생하면 기본적으로 스레드가 종료되는데, 이를 억제하고(Unwind), 에러를 우아하게(Gracefully) 처리하여 시스템의 가용성을 유지하는 것이 필수적입니다.

### 해결책: Catch Unwind와 모니터링 래퍼

Rust의 `std::panic::catch_unwind` 기능을 사용하여, 에이전트의 실행 로직을 감싸주는 래퍼(Wrapper)를 구현했습니다. 이를 통해 특정 에이전트에서 심각한 오류가 발생하더라도 런타임 자체는 살아남을 수 있습니다.

다음은 ZeroClaw 코어 라이브러리에 구현된 `AgentExecutor`의 간소화된 예제 코드입니다.

#### 1. 에이전트 트레이트 정의

먼저, 모든 에이전트가 따라야 할 공통 인터페이스를 정의합니다.

```rust
// src/agent/mod.rs

pub trait Agent: Send + Sync {
    /// 에이전트의 고유 ID를 반환합니다.
    fn id(&self) -> &str;

    /// 에이전트의 주요 실행 로직입니다.
    /// Result를 반환하여 일반적인 에러는 처리할 수 있도록 합니다.
    fn run(&self, input: Input) -> Result<Output, AgentError>;
}
```

#### 2. 안전한 실행 래퍼 구현 (Safe Executor)

이제 실제로 `Agent`를 실행하는 런타임 로직을 작성합니다. 여기서 `catch_unwind`를 사용하여 패닉을 잡아냅니다.

```rust
// src/runtime/executor.rs
use std::panic;
use std::any::Any;
use log::{error, info, warn};

pub struct SafeExecutor;

impl SafeExecutor {
    pub async fn execute<A>(agent: A, input: Input)
    where
        A: Agent + 'static,
    {
        let agent_id = agent.id().to_string();
        info!("[Executor] Agent {} 시작", agent_id);

        // catch_unwind는 힙에 할당된 데이터만 잡아낼 수 있으므로,
        // 에이전트 상태가 필요한 경우 AssertUnwindSafe를 사용해야 할 수 있습니다.
        // 여기서는 간단한 ID만 사용하므로 기본 컨텍스트로 진행합니다.
        let result = panic::catch_unwind(panic::AssertUnwindSafe(|| {
            // 실제 비즈니스 로직 실행 (동기 예시)
            agent.run(input)
        }));

        match result {
            Ok(output_result) => {
                match output_result {
                    Ok(output) => {
                        info!("[Executor] Agent {} 완료: {:?}", agent_id, output);
                    }
                    Err(e) => {
                        // 일반적인 로직 에러 (예: 파싱 실패, 통신 오류)
                        warn!("[Executor] Agent {} 로직 에러: {}", agent_id, e);
                        // 여기서 재시도 로직이나 DLQ(Dead Letter Queue) 전송 로직을 추가할 수 있습니다.
                    }
                }
            }
            Err(_) => {
                // 패닉이 발생하여 스택이 풀린 경우 (Panic Caught)
                let payload = e.downcast::<String>().map(|s| *s).unwrap_or_else(|_| "Unknown panic".to_string());
                
                error!("[Executor] Agent {} 패닉 발생: {}", agent_id, payload);
                // 
                // 중요: 이 시점에서 시스템은 종료되지 않았습니다.
                // Cloud Monitor와 연동하여 알림을 보내거나,
                // 다른 정상 에이전트에게 작업을 위임할 수 있습니다.
                //
            }
        }
    }
}
```

#### 3. 비동기 런타임(Tokio)과의 통합

위의 코드는 동기 실행을 예로 들었지만, ZeroClaw는 고성능을 위해 `tokio::task::spawn`을 사용합니다. 비동기 환경에서 `catch_unwind`를 사용할 때는 주의가 필요합니다.

```rust
// src/runtime/scheduler.rs
use tokio::task::JoinError;

pub async fn spawn_agent_task(agent: impl Agent + 'static, input: Input) {
    let agent_id = agent.id().to_string();

    let handle = tokio::task::spawn_blocking(move || {
        // CPU 연산이 많은 작업이거나 패닉 위험이 있는 작업은
        // spawn_blocking 내부에서 실행하는 것이 안전할 수 있습니다.
        // 또한 async fn 내부에서 panic이 발생할 경우를 대비해
        // Task 내부에서도 catch_unwind를 중첩하여 사용하는 것을 권장합니다.
        SafeExecutor::execute(agent, input)
    });

    // Task 자체가 실패했을 경우(취소 등)를 처리하기 위한 에러 핸들링
    match handle.await {
        Ok(_) => {},
        Err(e) if e.is_panic() => {
            error!("[Scheduler] Task {} panicked internally", agent_id);
        }
        Err(e) => {
            error!("[Scheduler] Task {} cancelled or aborted: {}", agent_id, e);
        }
    }
}
```

### 적용 사례 및 효과

이 구조를 적용한 후, [blog-api-server]의 LLM 호출 모듈에서 발생하던 예기치 못한 타임아웃 오류가 더 이상 전체 서버를 다운시키지 않게 되었습니다. 또한 [Discord Decision MCP] 아키텍처에서 복잡한 의사결정 트리를 수행하다가 하나의 프로세스가 실패해도, 다른 프로세스는 정상적으로 실행되어 전체 작업의 무결성을 보장할 수 있었습니다.
