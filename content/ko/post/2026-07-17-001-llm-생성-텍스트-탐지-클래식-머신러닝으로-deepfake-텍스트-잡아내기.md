# LLM 생성 텍스트 탐지: 클래식 머신러닝으로 'Deepfake' 텍스트 잡아내기

최근 AI 모델(Kimi, LM Studio 등)의 발전으로 LLM이 생성한 텍스트의 품질이 인간의 작성물과 거의 구별되지 않는 수준에 이르렀습니다. NotebookLM이 Gemini Notebook으로 통합되는 등 AI 도구가 우리의 일상 writing에 깊숙이 들어오고 있는 지금, "이 텍스트가 AI가 쓴 것인가?"라는 질문은 단순한 호기심을 넘어 보안과 진실성을 위한 중요한 문제가 되었습니다.

최근 Hacker News에 오른 'Detecting LLM-Generated Texts with Classical Machine Learning' 아티클은 거대한 모델을 사용하지 않고도, 전통적인 머신러닝(Classical ML) 기법으로 LLM 생성 텍스트를 효과적으로 탐지할 수 있음을 보여주었습니다. 이번 포스트에서는 이 방식을 직접 구현하여, 가볍고 효율적인 탐지 시스템을 구축하는 방법을 소개합니다.

## 왜 '클래식' 머신러닝인가?

LLM 탐지에 GPT-4와 같은 거대 모델을 사용하는 것은 '제비로 황새 잡기' 격일 수 있습니다. 높은 비용과 지연 시간(Latency)은 실시간 모니터링 시스템(예: [Cloud Monitor] 또는 [blog-api-server]의 로그 분석)에 치명적입니다.

반면, 랜덤 포레스트(Random Forest)나 SVM 같은 전통적인 알고리즘은 다음과 같은 장점을 가집니다.

1. **낮은 비용**: GPU가 필요 없으며 단일 CPU 코어에서도 빠르게 추론 가능.
2. **해석 가능성**: 모델이 왜 그렇게 판단했는지 설명하기 쉬움.
3. **배포의 용이성**: 모델 크기가 작아 엣지 디바이스나 레거시 서버에도 쉽게 통합.

이 접근법의 핵심은 LLM이 가진 **'Perplexity(혼란도)'**와 **'Burstiness(돌발성)'** 같은 통계적 패턴을 포착하는 것입니다.

## 탐지 시스템 아키텍처

실무 적용을 위해 간단한 파이프라인을 설계해 보겠습니다. 이 구조는 [ZeroClaw] 에이전트 런타임 내부에서 작동하는 가드레일(Guardrail) 컴포넌트로도 활용할 수 있습니다.

1. **데이터 전처리**: 텍스트를 토큰화하고 n-gram 등의 통계적 특징을 추출.
2. **특징 추출 (Feature Extraction)**: 단어 빈도, 문장 길이, 어휘 다양성 등의 정량적 지표 계산.
3. **분류 (Classification)**: 학습된 Random Forest 모델을 통해 "AI 생성" 또는 "인간 작성" 여부 판단.

## 실전 코드 예제 (Python)

Scikit-learn을 사용하여 간단한 탐지 모델을 구축해 보겠습니다. 실제 프로덕션 환경에서는 더 많은 데이터가 필요하지만, 여기서는 핵심 로직을 이해하는 데 집중하겠습니다.

### 1. 데이터 준비 및 특징 추출

LLM 텍스트는 보통 인간 텍스트보다 **'Perplexity(혼란도)'**가 낮고 문장 구조가 더 균일합니다. 이를 수치화하는 함수를 작성합니다.

```python
import re
import numpy as np
from collections import Counter

def extract_features(text):
    # 1. 평균 문장 길이
    sentences = re.split(r'[.!?]', text)
    avg_sentence_length = np.mean([len(s.split()) for s in sentences if s.strip()])

    # 2. 어휘 다양성 (Unique words / Total words)
    words = re.findall(r'\w+', text.lower())
    vocab_richness = len(set(words)) / len(words) if words else 0

    # 3. 문장 부호 빈도 (LLM은 종종 문장 부호 패턴이 다름)
    punct_count = len(re.findall(r'[,;:]', text))
    
    # 4. 불필요한 반복 확인 (Burstiness 척도 단순화)
    word_counts = Counter(words)
    most_common_count = word_counts.most_common(1)[0][1] if word_counts else 0
    repetition_ratio = most_common_count / len(words) if words else 0

    return [avg_sentence_length, vocab_richness, punct_count, repetition_ratio]
```

### 2. 모델 학습 및 예측

기존에 수집한 데이터셋이 있다고 가정하고, `RandomForestClassifier`를 학습시킵니다.

```python
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split

# 가상의 학습 데이터 (실제로는 충분한 데이터셋 필요)
# Feature: [평균 문장 길이, 어휘 다양성, 문장 부호 수, 반복 비율]
X = np.array([
    [15.0, 0.6, 5, 0.05], # 인간 작성 특성 (가상)
    [18.5, 0.65, 8, 0.04],
    [12.0, 0.5, 3, 0.02], 
    [25.0, 0.4, 2, 0.01], # AI 작성 특성 (가상) - 문장이 길고 어휘가 반복됨
    [28.0, 0.35, 1, 0.01],
])
y = np.array([0, 0, 0, 1, 1]) # 0: Human, 1: AI

# 모델 학습
clf = RandomForestClassifier(n_estimators=100, random_state=42)
clf.fit(X, y)

def is_ai_generated(text):
    features = [extract_features(text)]
    prediction = clf.predict(features)
    probability = clf.predict_proba(features)
    
    return "AI Generated" if prediction[0] == 1 else "Human Written", probability[0]
```

### 3. 실행 결과

이제 새로운 텍스트에 대해 판별을 내릴 수 있습니다.

```python
# 테스트 텍스트 (LLM 스타일)
test_text = """
Artificial intelligence is transforming the world. Technology is advancing rapidly. 
We must consider the implications of progress. The future is bright for innovation.
"""

label, prob = is_ai_generated(test_text)
print(f"Result: {label}")
print(f"Confidence: {prob}")
```

## 한계 및 보완 방안

이 '클래식' 방식은 완벽하지 않습니다. LLM이 "Perplexity"를 의도적으로 조작하는 Adversarial Attacks에 취약할 수 있습니다. 따라서 실제 시스템에서는 다음과 같은 하이브리드 접근이 권장됩니다.

1. **로그 분석과의 결합**: [blog-api-server] 로깅 개선 사례처럼, 사용자의 행동 패턴(Context)과 텍스트 특성을 함께 분석하면 정확도가 높아집니다.
2. **MCP 도구 활용**: [Discord MCP]나 [Cloud Monitor]와 연동하여 의심스러운 텍스트 생성 시도가 발견되면 즉시 알림을 보내는 워크플로우를 구축하세요.
3. **Zero-Shot 분류자**: 별도 학습 없이 LLM 자체를 이용해 "이 텍스트가 AI에 의해 쓰였는지 판단해"라고 물어보는 1차 필터링 단계를 추가하는 방법도 있습니다.
